## 1. 全体アーキテクチャ概要

本プロジェクトは **モノレポ**（1つのGitリポジトリに複数のアプリを収めた構成）です。  
以下の図は、ユーザーのブラウザからデータベースまでの全体的な流れを示しています。

```
                 ┌────────────────────────────────────────────────┐
                 │               ブラウザ (React)                 │
                 └──────┬──────────────────────┬─────────────────┘
                        │                      │
            管理画面アクセス            公開ページアクセス
                        │                      │
                        ▼                      ▼
           ┌──────────────────┐    ┌───────────────────┐
           │  admin-site      │    │  public-site      │
           │  (Next.js 16)    │    │  (Next.js SSG)    │
           │  CF Pages        │    │  CF Pages         │
           └────────┬─────────┘    └────────┬──────────┘
                    │ /api/*                │ fetch
                    ▼                       ▼
           ┌──────────────────────────────────────────┐
           │     admin-api-proxy (CF Worker)          │
           │     admin.shirasame.com/api/*             │
           └────────────────┬─────────────────────────┘
                            │ プロキシ転送
                            ▼
           ┌──────────────────────────────────────────┐
           │       public-worker (Hono on CF Worker)  │
           │       メインAPI サーバー                  │
           └──────┬───────────────────┬───────────────┘
                  │                   │
                  ▼                   ▼
         ┌──────────────┐    ┌──────────────────┐
         │  Supabase    │    │  Cloudflare R2   │
         │  (PostgreSQL │    │  (画像ストレージ) │
         │  + Auth)     │    │  + Image Resizing│
         └──────────────┘    └──────────────────┘
```

### 用語解説

| 用語 | 意味 |
|------|------|
| **モノレポ** | 複数のアプリやライブラリを 1 つのリポジトリで管理する手法 |
| **CF Pages** | Cloudflare Pages。静的サイトや Next.js アプリをホスティングするサービス |
| **CF Worker** | Cloudflare Workers。サーバーレスでJavaScript/TypeScriptを実行する環境 |
| **SSG** | Static Site Generation（静的サイト生成）。ビルド時にHTMLを事前生成する |
| **プロキシ** | リクエストを中継して別のサーバーに転送する仕組み |

## 2. フロントエンド

### 2.1 Next.js 16（App Router）

| 項目 | 内容 |
|------|------|
| **バージョン** | 16.0.0 |
| **公式サイト** | https://nextjs.org/ |
| **用途** | 管理画面 (`admin-site`) と公開サイト (`public-site`) の両方 |
| **選定理由** | React ベースのフルスタックフレームワーク。ルーティング・SSR・SSG・API Routes すべてを統合的に扱える |

```
💡 初心者向け解説
───────────────
Next.js は React をベースとした「フレームワーク」です。
React だけだと自分でルーティング（URL に応じたページ切り替え）や
サーバーサイドレンダリングの仕組みを作る必要がありますが、
Next.js はそれらを最初から備えています。

「App Router」は Next.js 13 以降で導入された新しいルーティング方式で、
`app/` フォルダの中のファイル構造がそのまま URL になります。
例: app/admin/products/page.tsx → /admin/products
```

#### 実際の使われ方

```
admin-site/app/
├── admin/
│   ├── page.tsx              ← /admin（ダッシュボード）
│   ├── products/page.tsx     ← /admin/products（商品一覧）
│   └── recipes/page.tsx      ← /admin/recipes（レシピ一覧）
├── api/
│   ├── products/route.ts     ← /api/products（API エンドポイント）
│   └── auth/login/route.ts   ← /api/auth/login（ログイン API）
└── layout.tsx                ← 全ページ共通レイアウト
```

## 3. バックエンド / サーバーサイド

### 3.1 Cloudflare Workers + Hono

| 項目 | 内容 |
|------|------|
| **Workers** | Cloudflare のサーバーレス実行環境 |
| **Hono** | Workers 向けの軽量 Web フレームワーク |
| **用途** | `public-worker/` — メインの公開 API サーバー |
| **選定理由** | 無料枠で高速。Express 風の書き方ができ学習コストが低い |

```
💡 初心者向け解説
───────────────
通常のWebアプリでは、サーバー（Express/Nest.js など）を
自分で管理する VPS やクラウド VM で動かします。
Cloudflare Workers は「サーバーレス」と呼ばれ、
サーバーの管理が不要で、リクエストが来たときだけコードが実行されます。
Hono は、その Workers 環境で Express 風にルーティングを書くための
軽量フレームワークです。
```

#### コード例（public-worker のルート定義）

```typescript
// public-worker/src/routes/public/products.ts のイメージ
import { Hono } from 'hono';
import { zValidator } from '@hono/zod-validator';
import { z } from 'zod';

const app = new Hono();

// GET /api/products — 商品一覧取得
app.get('/api/products', zValidator('query', z.object({
  limit: z.string().optional(),
  offset: z.string().optional(),
  tag: z.string().optional(),
})), async (c) => {
  // Supabase から商品データを取得
  const { data, error } = await supabase
    .from('products')
    .select('id, title, slug, price, published')
    .eq('published', true);

  return c.json({ data });
});
```

|------|------|

### 5.2 Cloudflare Image Resizing

| 項目 | 内容 |
|------|------|
| **用途** | R2 に保存した画像をオンデマンドでリサイズ・最適化して配信 |
| **選定理由** | 事前に複数サイズの画像を生成する必要がなく、URL パラメータで動的に変換可能 |

#### 画像 URL の生成パターン

```typescript
// shared/lib/image-usecases.ts より
// R2 キーから公開 URL を生成
function r2KeyToPublicUrl(key: string): string {
  return `https://images.shirasame.com/${key}`;
}

// Cloudflare Image Resizing の変換 URL を生成
// 例: https://images.shirasame.com/cdn-cgi/image/width=400,format=auto,quality=75/products/xxx/yyy.jpg
function buildResizingUrl(publicUrl: string, width: number): string {
  const base = 'https://images.shirasame.com';
  return `${base}/cdn-cgi/image/width=${width},format=auto,quality=75/${publicUrl}`;
}
```

#### 許可されている変換幅

```typescript
// コスト最適化のため、使用する幅を固定している
const ALLOWED_WIDTHS = [200, 400, 800, 1200, 1600, 1920];
```

### 画像アップロードの流れ

```
1. 管理画面でユーザーが画像をトリミング
        │
2. POST /api/images/direct-upload
   → R2 の Presigned URL を取得
        │
3. ブラウザから R2 に直接アップロード
        │
4. POST /api/images/complete
   → アップロード完了を通知、DB に画像メタ情報を保存
        │
5. 表示時: Image Resizing URL でオンデマンド変換
   https://images.shirasame.com/cdn-cgi/image/width=400,format=auto,quality=75/{key}
```

## 6. 認証 (Authentication)

### 6.1 Supabase Auth + HttpOnly Cookie

| 項目 | 内容 |
|------|------|
| **方式** | メールアドレス + パスワード認証（Supabase Auth） |
| **トークン管理** | HttpOnly Cookie + localStorage ミラー |
| **選定理由** | Supabase Auth はユーザー管理を丸ごと任せられる。Cookie による自動送信で API プロキシとの相性が良い |

```
💡 初心者向け解説
───────────────
「認証」とは、ユーザーが本人であることを確認する仕組みです。
ログインするとサーバーから「アクセストークン」と「リフレッシュトークン」が
発行されます。

- アクセストークン: API にリクエストする際の「身分証明書」。有効期限が短い（1時間程度）
- リフレッシュトークン: アクセストークンの期限が切れたとき、新しいものを取得するための鍵
- HttpOnly Cookie: JavaScript からアクセスできないCookie。XSS攻撃に強い
```

#### 認証フロー図

```
┌──────────┐        ┌─────────────┐        ┌─────────────┐
│ ブラウザ  │        │  admin-site │        │  Supabase   │
│          │        │  (サーバー)  │        │  Auth       │
└────┬─────┘        └──────┬──────┘        └──────┬──────┘
     │  1. ログイン情報送信  │                      │
     │ ────────────────────>│                      │
     │                      │  2. 認証リクエスト     │
     │                      │ ────────────────────>│
     │                      │  3. トークン返却       │
     │                      │ <────────────────────│
     │  4. Cookie設定 +      │                      │
     │     トークン返却      │                      │
     │ <────────────────────│                      │
     │                      │                      │
     │  5. 以降のAPI呼び出し │                      │
     │     (Cookie自動送信)  │                      │
     │ ────────────────────>│                      │
```
