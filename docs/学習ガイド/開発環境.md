## 1. 必要な前提ソフトウェア

### 1.1 インストールが必要なもの

| ソフトウェア | バージョン | 用途 | インストール方法 |
|------------|-----------|------|----------------|
| **Node.js** | ≥ 22.16.0 | JavaScript 実行環境 | https://nodejs.org/ |
| **pnpm** | 8.7.0 | パッケージマネージャー | `npm install -g pnpm@8.7.0` |
| **Git** | 最新 | バージョン管理 | https://git-scm.com/ |
| **VS Code** | 最新 | コードエディタ | https://code.visualstudio.com/ |
| **Wrangler** | 最新 | Cloudflare Workers CLI | `npm install -g wrangler` |

```
💡 初心者向け：各ツールの役割
──────────────────────────
Node.js:
  JavaScriptをパソコン上で動かすためのソフトウェア。
  ブラウザの外でもJavaScriptが使えるようになります。

pnpm:
  プロジェクトが必要とするライブラリ（依存パッケージ）を
  インストール・管理するツール。npm の高速版です。

Git:
  コードの変更履歴を記録するツール。
  「いつ・誰が・何を変更したか」を追跡できます。

Wrangler:
  Cloudflare Workers をローカルで開発・テスト・デプロイ
  するためのコマンドラインツール。
```

## 2. 初回セットアップ手順

### 2.1 手順の全体図

```
Step 1: リポジトリのクローン
    │
Step 2: 依存パッケージのインストール
    │
Step 3: 環境変数の設定
    │
Step 4: Supabase のセットアップ（初回のみ）
    │
Step 5: 開発サーバーの起動
    │
Step 6: ブラウザで確認 → 開発開始！
```

### 2.2 Step 1: リポジトリのクローン

```powershell
# プロジェクトをローカルにコピー
git clone https://github.com/your-username/shirasame-v0.git

# プロジェクトフォルダに移動
cd shirasame-v0
```

### 2.3 Step 2: 依存パッケージのインストール

```powershell
# ──────────────────────────────────────
# ルートの依存パッケージをインストール
# ──────────────────────────────────────
pnpm install

# ──────────────────────────────────────
# admin-site の依存パッケージをインストール
# ──────────────────────────────────────
cd admin-site
pnpm install
cd ..

# ──────────────────────────────────────
# public-site の依存パッケージをインストール
# ──────────────────────────────────────
cd public-site
pnpm install
cd ..

# ──────────────────────────────────────
# public-worker の依存パッケージをインストール
# ──────────────────────────────────────
cd public-worker
pnpm install
cd ..
```

```
💡 pnpm install とは？
────────────────────
package.json に書かれているライブラリを
自動的にダウンロードして node_modules/ に配置します。

初回は時間がかかりますが、2回目以降は
pnpm のキャッシュ機能により高速です。
```

### 2.4 Step 3: 環境変数の設定

```powershell
# admin-site の環境変数ファイルを作成
# テンプレートを元にコピー（テンプレートがない場合は手動作成）
cd admin-site
# .env.local ファイルを新規作成
```

`.env.local` ファイルの内容（最小構成）:

```env
# ── Supabase 設定（必須）──────────────────────
SUPABASE_URL=https://your-project-id.supabase.co
NEXT_PUBLIC_SUPABASE_URL=https://your-project-id.supabase.co
SUPABASE_ANON_KEY=your-anon-key
NEXT_PUBLIC_SUPABASE_ANON_KEY=your-anon-key
SUPABASE_SERVICE_ROLE_KEY=your-service-role-key

# ── 画像配信（必須）──────────────────────────
IMAGES_DOMAIN=https://images.shirasame.com
NEXT_PUBLIC_IMAGES_DOMAIN=https://images.shirasame.com

# ── R2 ストレージ（画像アップロード時に必要）───
R2_ACCOUNT=your-cloudflare-account-id
R2_ACCESS_KEY=your-r2-access-key
R2_SECRET=your-r2-secret-key
R2_BUCKET=images
R2_PUBLIC_URL=https://images.shirasame.com
NEXT_PUBLIC_R2_PUBLIC_URL=https://images.shirasame.com

# ── サイト設定 ─────────────────────────────
NEXT_PUBLIC_SITE_URL=http://localhost:3000
PUBLIC_PROFILE_EMAIL=your-owner-email@example.com
```

```
⚠️ 環境変数の値の取得方法
──────────────────────
Supabase:
  1. https://supabase.com にログイン
  2. プロジェクトを選択
  3. Settings → API → Project URL / anon key / service_role key

Cloudflare:
  1. https://dash.cloudflare.com にログイン
  2. R2 → Manage R2 API Tokens で Access Key/Secret を発行
  3. アカウント ID はダッシュボードの URL に表示される
```

### 2.5 Step 4: Supabase のセットアップ

```
💡 Supabase のセットアップが必要な場合
──────────────────────────────────
新しい Supabase プロジェクトを作成した場合のみ、
テーブルの作成が必要です。既にセットアップ済みの
プロジェクトに接続する場合はこのステップは不要です。
```

Supabase ダッシュボード → SQL Editor で以下を実行:

```sql
-- 基本テーブルの作成（簡略版）
-- 実際のマイグレーションは migrations/ フォルダ内の SQL を参照

-- ユーザーテーブル
CREATE TABLE IF NOT EXISTS users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email TEXT UNIQUE NOT NULL,
  display_name TEXT,
  avatar_url TEXT,
  bio TEXT,
  header_image_keys TEXT[],
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()
);

-- 商品テーブル
CREATE TABLE IF NOT EXISTS products (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(id),
  title TEXT NOT NULL,
  slug TEXT UNIQUE,
  price INTEGER DEFAULT 0,
  published BOOLEAN DEFAULT false,
  tags JSONB DEFAULT '[]'::jsonb,
  short_description TEXT,
  body TEXT,
  show_price BOOLEAN DEFAULT true,
  notes TEXT,
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()
);

-- 商品画像テーブル
CREATE TABLE IF NOT EXISTS product_images (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  product_id UUID NOT NULL REFERENCES products(id) ON DELETE CASCADE,
  image_key TEXT NOT NULL,
  width INTEGER,
  height INTEGER,
  role TEXT DEFAULT 'main',
  sort_order INTEGER DEFAULT 0,
  created_at TIMESTAMPTZ DEFAULT now()
);

-- その他のテーブルは migrations/ フォルダの SQL を順に実行
```

## 3. 開発サーバーの起動

### 3.1 admin-site（管理画面）

```powershell
# admin-site フォルダに移動
cd admin-site

# 開発サーバーを起動
pnpm dev
```

```
起動後のアクセス URL:
  http://localhost:3000         ← トップページ
  http://localhost:3000/admin   ← 管理画面
  http://localhost:3000/admin/login ← ログイン画面
```

```
💡 pnpm dev の動作
────────────────
Next.js の開発サーバーが起動します。
ファイルを編集すると、自動的にブラウザが更新されます
（Hot Module Replacement = HMR）。
コードを保存するたびにリロードする必要はありません。

開発サーバーではプロキシが無効になるため、
admin-site/app/api/ 内の API Routes が直接使用されます。
```

### 3.2 public-worker（API サーバー）

```powershell
# public-worker フォルダに移動
cd public-worker

# ローカル開発サーバーを起動（Miniflare）
npx wrangler dev
```

```
起動後のアクセス URL:
  http://localhost:8787         ← Worker のローカルサーバー
  http://localhost:8787/api/products ← 商品 API

Miniflare:
  wrangler dev は「Miniflare」というローカルエミュレータを起動します。
  Cloudflare Workers の動作をローカルで再現できます。
  R2 バケットもローカルでエミュレートされます。
```

### 3.3 public-site（公開サイト）

```powershell
# public-site フォルダに移動
cd public-site

# 開発サーバーを起動
pnpm dev
```

```
起動後のアクセス URL:
  http://localhost:3001    ← 公開サイト（ポートが異なる場合あり）
```

### 3.4 全体の開発構成

```
┌─────────────────────────────────────────────────────────┐
│                   ローカル開発環境                        │
│                                                         │
│  ターミナル1:                                            │
│    cd admin-site && pnpm dev                            │
│    → http://localhost:3000  （管理画面 + API）            │
│                                                         │
│  ターミナル2（オプション）:                               │
│    cd public-worker && npx wrangler dev                  │
│    → http://localhost:8787  （Workers API）               │
│                                                         │
│  ターミナル3（オプション）:                               │
│    cd public-site && pnpm dev                            │
│    → http://localhost:3001  （公開サイト）                │
│                                                         │
│  ※ 通常の開発では ターミナル1 だけで十分です              │
│    （admin-site 内に API Routes があるため）              │
└─────────────────────────────────────────────────────────┘
```

## 4. ビルド手順

### 4.1 admin-site のビルド

```powershell
# プロジェクトルートから
pnpm run build:admin

# または admin-site フォルダ内で
cd admin-site
pnpm run build
```

```
💡 ビルドとは？
────────────
TypeScript → JavaScript への変換、
React コンポーネントの最適化、
CSS の圧縮、不要なコードの削除などを行い、
本番環境で動作する最適化されたファイルを生成します。

ビルド成果物:
  admin-site/.next/    ← ビルドされたファイル群
```

### 4.2 public-site のビルド

```powershell
# プロジェクトルートから
pnpm run build:public

# 内部で実行されるコマンド:
# cd public-site && pnpm install && pnpm run build:pages
```

```
💡 SSG ビルドの仕組み
──────────────────
public-site は SSG（Static Site Generation）モードでビルドします。
ビルド時に HTML ファイルを事前生成し、
Cloudflare Pages にデプロイします。

ビルド成果物:
  public-site/out/     ← 静的ファイル群（HTML/CSS/JS）
```

### 4.3 public-worker のビルド

```powershell
cd public-worker

# デプロイ（ビルド + アップロード）
npx wrangler deploy
```

### 4.4 ビルドコマンド一覧

| コマンド | 場所 | 説明 |
|---------|------|------|
| `pnpm run build` | ルート | admin-site をビルド |
| `pnpm run build:admin` | ルート | admin-site をビルド |
| `pnpm run build:public` | ルート | public-site をビルド |
| `pnpm dev` | admin-site | 開発サーバー起動 |
| `pnpm start` | admin-site | 本番サーバー起動（ビルド後） |
| `npx wrangler dev` | public-worker | ローカル Workers 起動 |
| `npx wrangler deploy` | public-worker | Workers デプロイ |

## 5. テストの実行

### 5.1 E2E テスト（Playwright）

```powershell
# ──────────────────────────────────────
# Playwright のインストール（初回のみ）
# ──────────────────────────────────────
npx playwright install

# ──────────────────────────────────────
# E2E テストの実行
# ──────────────────────────────────────
# プロジェクトルートから
pnpm run test:e2e

# または直接 Playwright を実行
npx playwright test tests/e2e --config=playwright.config.ts
```

```
💡 E2E テストとは？
────────────────
End-to-End（端から端まで）テストの略。
実際のブラウザを起動して、ユーザーの操作をシミュレートし、
アプリケーション全体が正しく動作するかを確認します。

例:
  1. ログインページを開く
  2. メールアドレスとパスワードを入力
  3. ログインボタンをクリック
  4. ダッシュボードに遷移したことを確認
```

### 5.2 テストの設定

```typescript
// playwright.config.ts
import { defineConfig } from '@playwright/test';

export default defineConfig({
  timeout: 30_000,          // テストのタイムアウト: 30秒
  use: {
    headless: true,          // ブラウザを画面表示せずに実行
    baseURL: process.env.STAGING_BASE_URL || 'http://localhost:3000',
  },
  projects: [
    { name: 'chromium' },    // Chromium ブラウザでテスト
  ],
});
```

### 5.3 テスト実行に必要な環境変数

```env
# テスト用の認証情報
TEST_EMAIL=test-user@example.com
TEST_PASSWORD=test-password
STAGING_BASE_URL=http://localhost:3000
```

### 5.4 テストの書き方

```typescript
// tests/e2e/admin-auth.spec.ts
import { test, expect } from '@playwright/test';

test.describe('管理画面の認証', () => {

  test('未ログインでは管理画面にアクセスできない', async ({ page }) => {
    // 管理画面にアクセス
    await page.goto('/admin');

    // ログインページにリダイレクトされることを確認
    await expect(page).toHaveURL(/\/admin\/login/);
  });

  test('正しい認証情報でログインできる', async ({ page }) => {
    await page.goto('/admin/login');

    // フォームに入力
    await page.fill('[name=email]', process.env.TEST_EMAIL!);
    await page.fill('[name=password]', process.env.TEST_PASSWORD!);

    // ログインボタンをクリック
    await page.click('button[type=submit]');

    // ダッシュボードに遷移したことを確認
    await expect(page).toHaveURL(/\/admin/);
    await expect(page.locator('h1')).toContainText('ダッシュボード');
  });

});
```

### 5.5 手動テスト用スクリプト


## 8. 開発時のよくある作業フロー

### 8.1 新しいページを追加する

```
1. フォルダとファイルを作成
   admin-site/app/admin/新機能名/page.tsx

2. コンポーネントを実装
   - React コンポーネントとして画面を作成
   - 必要ならコンポーネントを components/ に分離

3. ナビゲーションにリンクを追加
   components/admin-nav.tsx にメニュー項目を追加

4. 開発サーバーで確認
   http://localhost:3000/admin/新機能名
```

### 8.2 新しい API エンドポイントを追加する

```
1. Route Handler を作成
   admin-site/app/api/新エンドポイント/route.ts

2. HTTP メソッドをエクスポート
   export async function GET(req) { ... }
   export async function POST(req) { ... }

3. テスト
   curl http://localhost:3000/api/新エンドポイント

4. （必要に応じて）Workers にも実装
   public-worker/src/routes/public/新エンドポイント.ts
```

### 8.3 DB スキーマを変更する

```
1. マイグレーション SQL を作成
   migrations/YYYY-MM-DD-変更内容.sql

2. Supabase ダッシュボードで SQL を実行
   SQL Editor → 新規クエリ → SQL を貼り付け → 実行

3. TypeScript の型定義を更新
   - admin-site/lib/db/schema.ts
   - shared/types/index.ts

4. 関連する API やコンポーネントを修正

5. テストで動作確認
```

### 8.4 コンポーネントを作成する

```typescript
// 1. ファイルを作成: admin-site/components/my-component.tsx

'use client'; // ブラウザで動作するコンポーネントに必須

import { useState } from 'react';
import { Button } from '@/components/ui/button';

interface MyComponentProps {
  title: string;
  onSubmit: (value: string) => void;
}

export function MyComponent({ title, onSubmit }: MyComponentProps) {
  const [value, setValue] = useState('');

  return (
    <div className="rounded-lg border p-4">
      <h2 className="text-lg font-bold">{title}</h2>
      <input
        className="mt-2 w-full rounded border px-3 py-2"
        value={value}
        onChange={(e) => setValue(e.target.value)}
      />
      <Button
        className="mt-2"
        onClick={() => onSubmit(value)}
      >
        送信
      </Button>
    </div>
  );
}
```

## 9. トラブルシューティング

### 9.1 よくあるエラーと対処法

| エラー | 原因 | 対処法 |
|--------|------|--------|
| `Module not found` | パッケージ未インストール | `pnpm install` を実行 |
| `SUPABASE_URL is not defined` | 環境変数未設定 | `.env.local` を確認 |
| `401 Unauthorized` | 認証トークン切れ | 再ログイン |
| `CORS error` | クロスオリジン問題 | 開発時は同一オリジンで確認 |
| `Port 3000 already in use` | 別プロセスがポート使用中 | プロセス停止 or 別ポート指定 |
| `pnpm: command not found` | pnpm 未インストール | `npm install -g pnpm` |

### 9.2 環境変数のトラブル

```powershell
# 環境変数が読み込まれているか確認
cd admin-site
node -e "console.log(process.env.SUPABASE_URL)"

# Next.js で NEXT_PUBLIC_ 変数を確認
# → 開発サーバーを再起動する必要がある（環境変数はビルド時に埋め込まれる）
```

### 9.3 依存パッケージのトラブル

```powershell
# ── キャッシュクリア + 再インストール ──────────
cd admin-site
Remove-Item -Recurse -Force node_modules
Remove-Item pnpm-lock.yaml
pnpm install

# ── pnpm のキャッシュクリア ──────────────────
pnpm store prune
```

### 9.4 ビルドエラー

```powershell
# ── TypeScript の型エラーを確認 ──────────────
cd admin-site
npx tsc --noEmit

# ── ESLint のエラーを確認 ────────────────────
pnpm lint

# ── .next キャッシュの削除 ───────────────────
Remove-Item -Recurse -Force .next
pnpm dev
```
