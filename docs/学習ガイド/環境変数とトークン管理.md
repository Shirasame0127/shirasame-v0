## 1. 環境変数とは？

```
💡 初心者向け解説
───────────────
環境変数とは、アプリケーションの外側から設定値を渡す仕組みです。

なぜ使うのか？
  ✅ パスワードやAPIキーをコードに直書きしない（セキュリティ）
  ✅ 開発環境と本番環境で異なる設定を簡単に切り替えられる
  ✅ 設定変更のたびにコードを変更する必要がない

例えるなら:
  環境変数 = アプリに渡す「設定カード」
  .env ファイル = 設定カードの束を入れた「封筒」
  Secrets = 金庫の中に入れた「最重要設定カード」
```

### 環境変数の優先度

```
  高い ◄────────────────────────────────────────────────── 低い
  
  Cloudflare Secrets  >  Cloudflare環境変数  >  .env.local  >  .env
  (暗号化・最重要)       (Workers/Pages設定)     (ローカル上書き)  (デフォルト)
```

## 2. 環境変数の全体一覧

### 2.1 Supabase 関連

| 変数名 | 機密性 | 使用場所 | 説明 |
|--------|--------|---------|------|
| `SUPABASE_URL` | 🟡 中 | admin-site / worker | Supabase プロジェクトの URL |
| `NEXT_PUBLIC_SUPABASE_URL` | 🟢 低 | admin-site（ブラウザ） | ブラウザ側で使う Supabase URL |
| `SUPABASE_ANON_KEY` | 🟡 中 | admin-site / worker | Supabase の匿名キー（RLS 制限あり） |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | 🟢 低 | admin-site（ブラウザ） | ブラウザ側で使う匿名キー |
| `SUPABASE_SERVICE_ROLE_KEY` | 🔴 高 | admin-site（サーバー） / worker | **管理者権限キー（全テーブル操作可能）** |

```
⚠️ 重要
───────
SUPABASE_SERVICE_ROLE_KEY は最も機密性の高いキーです。
これが漏洩すると、データベースの全データに無制限でアクセスされます。

絶対に:
  ❌ ブラウザ側のコードで使わない
  ❌ Git にコミットしない
  ❌ NEXT_PUBLIC_ プレフィックスを付けない
```

### 2.2 Cloudflare R2 関連（画像ストレージ）

| 変数名 | 機密性 | 使用場所 | 説明 |
|--------|--------|---------|------|
| `CLOUDFLARE_ACCOUNT_ID` / `R2_ACCOUNT` | 🟡 中 | admin-site | Cloudflare アカウント ID |
| `R2_ACCESS_KEY_ID` / `R2_ACCESS_KEY` | 🔴 高 | admin-site | R2 アクセスキー |
| `R2_SECRET_ACCESS_KEY` / `R2_SECRET` | 🔴 高 | admin-site | R2 シークレットキー |
| `R2_BUCKET` | 🟢 低 | admin-site / worker | R2 バケット名（デフォルト: `images`） |
| `NEXT_PUBLIC_R2_PUBLIC_URL` / `R2_PUBLIC_URL` | 🟢 低 | 全体 | R2 の公開エンドポイント URL |

### 2.3 CDN / 画像配信関連

| 変数名 | 機密性 | 使用場所 | 説明 |
|--------|--------|---------|------|
| `CDN_BASE_URL` / `NEXT_PUBLIC_CDN_BASE` | 🟢 低 | admin-site / worker | CDN のベース URL |
| `IMAGES_DOMAIN` / `NEXT_PUBLIC_IMAGES_DOMAIN` | 🟢 低 | 全体 | 画像配信ドメイン（例: `https://images.shirasame.com`） |
| `ALLOWED_IMAGE_HOSTS` | 🟢 低 | admin-site | サムネイル変換を許可するホスト（カンマ区切り） |

### 2.4 認証・ホスト関連

| 変数名 | 機密性 | 使用場所 | 説明 |
|--------|--------|---------|------|
| `PUBLIC_PROFILE_EMAIL` | 🟡 中 | admin-site / worker | 公開ページのオーナー特定用メールアドレス |
| `PUBLIC_HOST` / `NEXT_PUBLIC_PUBLIC_HOST` | 🟢 低 | admin-site | 公開サイトのホスト名 |
| `PUBLIC_OWNER_USER_ID` | 🟡 中 | public-site | 公開ページのオーナーユーザー ID |
| `NEXT_PUBLIC_SITE_URL` | 🟢 低 | admin-site / public-site | サイトの URL |

### 2.5 API 関連

| 変数名 | 機密性 | 使用場所 | 説明 |
|--------|--------|---------|------|
| `API_BASE_ORIGIN` / `NEXT_PUBLIC_API_BASE_URL` | 🟢 低 | admin-site | API のベース URL |
| `NEXT_PUBLIC_PUBLIC_API_BASE` | 🟢 低 | public-site | 公開 API のベース URL |
| `INTERNAL_API_BASE` | 🟡 中 | admin-site / worker | 内部 API のベース URL |
| `INTERNAL_API_KEY` | 🔴 高 | admin-site / worker | 内部 API の認証キー |
| `ADMIN_FORCE_EXTERNAL_API` | 🟢 低 | admin-site | 開発時に外部 API を強制使用するフラグ |

### 2.6 実行環境

| 変数名 | 機密性 | 使用場所 | 説明 |
|--------|--------|---------|------|
| `NODE_ENV` | 🟢 低 | 全体 | `development` / `production` |
| `NEXT_OUTPUT_EXPORT` | 🟢 低 | admin-site / public-site | `1` のとき SSG（静的出力）モード |

## 3. トークンの種類と役割

### 3.1 トークンの全体像

```
┌───────────────────────────────────────────────────────────┐
│                    トークンの種類                          │
├────────────────────┬──────────────────────────────────────┤
│  認証トークン       │  ユーザーのログイン状態を証明する     │
│  (Access Token)    │  有効期限: 約1時間                   │
├────────────────────┼──────────────────────────────────────┤
│  リフレッシュ       │  アクセストークンを再取得するための   │
│  トークン          │  鍵。有効期限: 長め（日〜週単位）     │
│  (Refresh Token)   │                                      │
├────────────────────┼──────────────────────────────────────┤
│  Anon Key          │  Supabase への公開用アクセスキー      │
│                    │  RLS で制限されたデータのみ操作可能    │
├────────────────────┼──────────────────────────────────────┤
│  Service Role Key  │  Supabase への管理者キー             │
│                    │  RLS をバイパスし全データ操作可能      │
├────────────────────┼──────────────────────────────────────┤
│  R2 Access Key     │  R2 ストレージへのアクセス認証情報    │
│  + Secret Key      │  S3 互換 API での認証に使用           │
├────────────────────┼──────────────────────────────────────┤
│  Internal API Key  │  内部サービス間通信の認証キー         │
│                    │  Worker 間の安全な通信に使用           │
└────────────────────┴──────────────────────────────────────┘
```

### 3.2 認証トークンの流れ

```
1. ログイン
   ユーザー → [メール/パスワード] → Supabase Auth
                                      │
                                      ▼
                              access_token (JWT)
                              refresh_token
                                      │
2. トークン保存                       ▼
   ┌─────────────────────────────────────────┐
   │  HttpOnly Cookie   │  localStorage      │
   │  sb-access-token   │  sb-access-token   │
   │  sb-refresh-token  │  (ミラーコピー)     │
   │                    │                     │
   │  ✅ XSS に強い     │  ✅ JS からアクセス  │
   │  ❌ JS 非アクセス   │  ❌ XSS に弱い     │
   └─────────────────────────────────────────┘

3. API リクエスト時
   ブラウザ → Cookie が自動送信される
            → Authorization ヘッダーにも設定

4. トークンリフレッシュ（15分ごと）
   ブラウザ → POST /api/auth/refresh
            → 新しい access_token を取得
            → Cookie と localStorage を更新
```

```
💡 初心者向け解説
───────────────
なぜ Cookie と localStorage の両方に保存するのか？

Cookie（HttpOnly）:
  → XSS 攻撃（悪意あるスクリプトの注入）から守れる
  → API プロキシ経由で自動送信されるので便利

localStorage:
  → JavaScript から直接トークンを読み取れる
  → Authorization ヘッダーに設定する際に必要

両方を使うことで、セキュリティと利便性のバランスを取っています。
```

### 3.3 Anon Key vs Service Role Key

```
┌─────────────────────┬──────────────────────────────────────┐
│      Anon Key       │        Service Role Key              │
├─────────────────────┼──────────────────────────────────────┤
│ ブラウザで使える     │ サーバーでのみ使う                   │
│ RLS でデータ制限     │ RLS をバイパス（全データアクセス可）  │
│ 漏洩しても被害は限定 │ 漏洩すると致命的                    │
│ NEXT_PUBLIC_ OK      │ NEXT_PUBLIC_ 絶対NG                 │
└─────────────────────┴──────────────────────────────────────┘

RLS（Row Level Security）とは？
───────────────────────────────
データベースの各行（レコード）ごとにアクセス権限を設定する仕組みです。
例: 「自分が作成した商品データのみ読み書きできる」

Anon Key は RLS のルールに従うので、
たとえブラウザに露出しても、許可されたデータしかアクセスできません。
```

## 4. 環境別の管理方法

### 4.1 管理方法の使い分け

| 環境 | 管理方法 | 対象 |
|------|---------|------|
| **ローカル開発** | `.env.local` ファイル | admin-site / public-site |
| **Cloudflare Pages** | ダッシュボードの環境変数設定 | admin-site / public-site |
| **Cloudflare Workers** | `wrangler.toml` + Secrets | public-worker / admin-api-proxy |
| **Supabase** | ダッシュボードの設定画面 | DB 接続情報・キー |

### 4.2 ローカル開発環境（.env ファイル）

```
💡 .env ファイルの種類
────────────────────
.env           → デフォルト値（Git に含めてOK — 機密情報なし）
.env.local     → ローカル上書き（Git に含めない！.gitignore で除外）
.env.example   → チームメンバー向けのテンプレート（Git に含める）
```

#### .env.local の配置場所

```
shirasame-v0/
├── admin-site/
│   └── .env.local           ← admin-site 用の環境変数
├── public-site/
│   └── .env.local           ← public-site 用の環境変数
└── public-worker/
    └── .dev.vars             ← Workers ローカル開発用（wrangler dev）
```

### 4.3 Cloudflare Pages の環境変数設定

```
Cloudflare ダッシュボード
  └── Pages
      └── プロジェクト選択
          └── Settings
              └── Environment Variables
                  ├── Production（本番環境用）
                  │   ├── SUPABASE_URL = https://xxx.supabase.co
                  │   ├── SUPABASE_SERVICE_ROLE_KEY = ••••••（暗号化）
                  │   └── ...
                  └── Preview（プレビュー環境用）
                      └── ...
```

### 4.4 Cloudflare Workers の Secrets

```bash
# wrangler CLI で Secrets を設定（暗号化されて保存される）
npx wrangler secret put SUPABASE_SERVICE_ROLE_KEY
# → プロンプトで値を入力

npx wrangler secret put INTERNAL_API_KEY
# → プロンプトで値を入力

# wrangler.toml には機密でない変数を記載
# [vars]
# IMAGES_DOMAIN = "https://images.shirasame.com"
# R2_BUCKET = "images"
```

```
⚠️ 重要な違い
─────────────
wrangler.toml の [vars]:
  → Git に含まれる → 機密情報は NG
  → 変更時は再デプロイが必要

wrangler secret:
  → 暗号化されて Cloudflare に保存
  → Git には含まれない → 機密情報は OK
  → ダッシュボードからも設定可能
```

## 6. コード内での環境変数の使い方

### 6.1 Next.js での環境変数アクセス

```typescript
// ─────────────────────────────────────────
// サーバー側（API Route / Server Component）
// ─────────────────────────────────────────
// process.env でアクセス可能
// NEXT_PUBLIC_ プレフィックスは不要
const supabaseUrl = process.env.SUPABASE_URL;
const serviceRoleKey = process.env.SUPABASE_SERVICE_ROLE_KEY;

// ─────────────────────────────────────────
// ブラウザ側（Client Component）
// ─────────────────────────────────────────
// NEXT_PUBLIC_ プレフィックスが必須！
const supabaseUrl = process.env.NEXT_PUBLIC_SUPABASE_URL;
const imagesDomain = process.env.NEXT_PUBLIC_IMAGES_DOMAIN;
```

```
💡 NEXT_PUBLIC_ プレフィックスとは？
──────────────────────────────────
Next.js では、ブラウザに送信される JavaScript に
環境変数の値を埋め込むことができます。

NEXT_PUBLIC_ で始まる変数:
  → ビルド時にブラウザ用の JS に埋め込まれる
  → ブラウザで process.env.NEXT_PUBLIC_XXX としてアクセスできる
  → ⚠️ 公開情報と見なされる（機密情報は絶対NG！）

NEXT_PUBLIC_ なしの変数:
  → サーバー側でのみアクセスできる
  → ブラウザの JS には含まれない
  → 機密情報はこちらで管理する
```

### 6.2 Supabase クライアントの初期化

```typescript
// ── ブラウザ用（Anon Key を使用）──────────────────────
// admin-site/lib/supabase/client.ts
import { createClient } from '@supabase/supabase-js';

const SUPABASE_URL = process.env.NEXT_PUBLIC_SUPABASE_URL!;
const SUPABASE_ANON_KEY = process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!;

export const supabase = createClient(SUPABASE_URL, SUPABASE_ANON_KEY, {
  auth: {
    persistSession: false,    // 自前でセッション管理
    autoRefreshToken: false,  // 自前でリフレッシュ
  },
});


// ── サーバー用（Service Role Key を使用）──────────────
// admin-site/lib/supabase.ts
import { createClient } from '@supabase/supabase-js';

export function getSupabaseAdmin() {
  const url = process.env.SUPABASE_URL;
  const key = process.env.SUPABASE_SERVICE_ROLE_KEY;

  if (!url || !key) {
    throw new Error('Supabase の環境変数が設定されていません');
  }

  return createClient(url, key, {
    auth: { persistSession: false },
  });
}
```

### 6.5 API クライアントでの認証ヘッダー設定

```typescript
// ── admin-site/lib/api-client.ts（簡略版）─────────
export async function apiFetch(path: string, init?: RequestInit) {
  const url = resolveApiUrl(path);

  // localStorage からトークンを取得
  const token = typeof window !== 'undefined'
    ? localStorage.getItem('sb-access-token')
    : null;

  const headers = new Headers(init?.headers);

  // 認証ヘッダーを自動設定
  if (token) {
    headers.set('Authorization', `Bearer ${token}`);
  }

  const response = await fetch(url, {
    ...init,
    headers,
    credentials: 'include',  // Cookie も自動送信
  });

  if (response.status === 401) {
    throw new Error('unauthenticated');
  }

  return response.json();
}
```

### 6.6 Workers 環境での環境変数

```typescript
// ── Cloudflare Workers（Hono）─────────────────────
// Workers では process.env ではなく、リクエストの context から取得

import { Hono } from 'hono';

type Bindings = {
  SUPABASE_URL: string;
  SUPABASE_SERVICE_ROLE_KEY: string;
  IMAGES_BUCKET: R2Bucket;       // R2 バケットのバインディング
  IMAGES_DOMAIN: string;
};

const app = new Hono<{ Bindings: Bindings }>();

app.get('/api/products', async (c) => {
  // c.env で環境変数にアクセス
  const supabaseUrl = c.env.SUPABASE_URL;
  const serviceKey = c.env.SUPABASE_SERVICE_ROLE_KEY;

  // R2 バケットに直接アクセス
  const object = await c.env.IMAGES_BUCKET.get('products/abc/001.jpg');

  // ...
});
```

```
💡 Workers での環境変数の違い
──────────────────────────
Node.js:   process.env.SUPABASE_URL
Workers:   c.env.SUPABASE_URL（Hono の場合）
           env.SUPABASE_URL（素の Workers の場合）

Workers では process.env は使えません。
代わりに、リクエストハンドラの引数から環境変数を取得します。
wrangler.toml の [vars] や Secrets で設定した値が
自動的に env オブジェクトに注入されます。
```

## 7. セキュリティのベストプラクティス

### 7.1 チェックリスト

| ✅ やるべきこと | ❌ やってはいけないこと |
|----------------|----------------------|
| `.env.local` は `.gitignore` に追加 | `.env.local` を Git にコミット |
| 機密キーは Cloudflare Secrets で管理 | `wrangler.toml` に機密キーを記載 |
| `NEXT_PUBLIC_` は公開して良い値のみ | Service Role Key に `NEXT_PUBLIC_` を付ける |
| R2 キーはサーバー側でのみ使用 | ブラウザから直接 R2 認証情報を使う |
| 定期的にキーをローテーション | 古いキーを放置 |

### 7.2 .gitignore の設定

```gitignore
# 環境変数ファイル（機密情報を含むため）
.env.local
.env.*.local
.dev.vars

# Cloudflare Workers
.wrangler/

# Node.js
node_modules/

# Next.js
.next/
```

### 7.3 キーが漏洩した場合の対応

```
1. Supabase Service Role Key が漏洩した場合:
   → Supabase ダッシュボード → Settings → API → Generate New Key
   → 全デプロイ環境で新しいキーに更新

2. R2 Access Key が漏洩した場合:
   → Cloudflare ダッシュボード → R2 → Manage R2 API Tokens
   → 古いトークンを無効化 → 新しいトークンを発行

3. Internal API Key が漏洩した場合:
   → 新しいランダム文字列を生成
   → 全 Workers と admin-site で更新
```

### 7.4 環境変数のバリデーション

```typescript
// ── 起動時に必須の環境変数をチェックする例 ───────────
function validateEnv() {
  const required = [
    'SUPABASE_URL',
    'SUPABASE_SERVICE_ROLE_KEY',
  ];

  const missing = required.filter(key => !process.env[key]);

  if (missing.length > 0) {
    throw new Error(
      `必須の環境変数が設定されていません: ${missing.join(', ')}\n` +
      `.env.local ファイルを確認してください。`
    );
  }
}

// アプリ起動時に呼び出す
validateEnv();
```

## クイックリファレンス（早見表）

### どの変数をどこで設定するか

```
ローカル開発:
  admin-site/.env.local       ← Supabase / R2 / API 設定
  public-site/.env.local      ← 公開 API / 画像ドメイン
  public-worker/.dev.vars     ← Workers ローカル開発用

本番 (Cloudflare Pages):
  ダッシュボード → Environment Variables
  ├── SUPABASE_URL
  ├── SUPABASE_SERVICE_ROLE_KEY  ← 「Encrypt」にチェック
  ├── NEXT_PUBLIC_IMAGES_DOMAIN
  └── ...

本番 (Cloudflare Workers):
  wrangler.toml [vars]        ← 公開値 (IMAGES_DOMAIN 等)
  wrangler secret put XXX     ← 機密値 (SERVICE_ROLE_KEY 等)
```
