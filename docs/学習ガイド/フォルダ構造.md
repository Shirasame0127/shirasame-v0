## 1. ルート構成の全体像

```
shirasame-v0/                    ← プロジェクトルート（モノレポ）
│
├── 📂 admin-site/               ← ★ 管理画面（Next.js 16 + React 19）
├── 📂 public-site/              ← ★ 公開サイト（Next.js SSG）
├── 📂 public-worker/            ← ★ 公開 API サーバー（Hono + CF Workers）
├── 📂 cloudflare/               ← Cloudflare Worker 群（プロキシ等）
│   ├── admin-api-proxy/         ←   管理サイト→API プロキシ
│   └── internal-api-worker/     ←   内部 API（JWT 検証）
├── 📂 shared/                   ← 複数アプリで共有する型・ユーティリティ
├── 📂 docs/                     ← プロジェクトドキュメント
├── 📂 migrations/               ← DB マイグレーション SQL
├── 📂 scripts/                  ← 運用・開発用スクリプト
├── 📂 tests/                    ← E2E テスト
├── 📂 v0-public/                ← （旧公開サイト残骸・参照用）
│
├── 📄 package.json              ← ルートのパッケージ定義・ビルドスクリプト
├── 📄 pnpm-lock.yaml            ← 依存関係のロックファイル
├── 📄 playwright.config.ts      ← E2E テスト設定
└── 📄 README.md                 ← プロジェクト概要
```

```
💡 初心者向け解説
───────────────
「モノレポ」とは、管理画面・公開サイト・API サーバーなど
複数のアプリケーションを 1 つの Git リポジトリで管理する手法です。

メリット:
  - コードの共有が簡単（shared/ フォルダでまとめられる）
  - 変更の影響範囲が一目でわかる
  - 1 回のプルリクエストで複数アプリの修正ができる

デメリット:
  - リポジトリが大きくなりがち
  - CI/CD の設定がやや複雑になる
```

### 2.2 重要ファイルの役割

#### `app/` — App Router のルール

```
💡 初心者向け解説
───────────────
Next.js App Router では、ファイルの配置場所がそのまま URL になります。

- app/admin/page.tsx         → /admin
- app/admin/products/page.tsx → /admin/products
- app/api/products/route.ts   → /api/products（API エンドポイント）

特殊なファイル名:
- page.tsx   : そのフォルダの URL にアクセスしたとき表示されるページ
- layout.tsx : 子ページすべてに適用される共通レイアウト
- route.ts   : API エンドポイント（GET/POST/PUT/DELETE を export）
- loading.tsx: ページ読み込み中に表示される UI
- error.tsx  : エラー発生時に表示される UI
```

#### `components/ui/` — UI コンポーネントライブラリ

```
💡 「shadcn/ui」パターンとは？
─────────────────────────────
shadcn/ui は、コンポーネントを npm パッケージとしてインストールするのではなく、
ソースコードを直接プロジェクトにコピーして使う方式のUIライブラリです。

メリット:
- コンポーネントのカスタマイズが自由自在
- 不要な依存関係が増えない
- プロジェクトに最適化しやすい

components/ui/ の中のファイルは基本的にこのパターンで配置されています。
Radix UI（アクセシビリティ対応のプリミティブ）+ Tailwind CSS + CVA で
構成されています。
```

#### `lib/` — レイヤー構造

```
lib/
├── auth.ts          ← 認証層: ログイン/ログアウト/セッション管理
├── api-client.ts    ← API層: URL ルーティング + 認証付き fetch
├── db/
│   ├── schema.ts    ← データ層: テーブルの型定義
│   └── storage.ts   ← データ層: キャッシュ + データ取得
└── services/        ← サービス層: ビジネスロジック

呼び出し順序:
  コンポーネント → lib/services/ → lib/api-client.ts → API → DB
```

#### `middleware.ts` — リクエストの門番

```typescript
// middleware.ts の処理フロー
//
// リクエスト
//     │
//     ├── /api/* にマッチ？
//     │   ├── YES → 外部 API（public-worker）へプロキシ
//     │   │        （開発時はそのまま Next.js 内部 API で処理）
//     │   └── NO ↓
//     │
//     ├── /admin/* にマッチ？
//     │   ├── YES → Cookie で認証チェック
//     │   │        ├── 認証OK → ページ表示
//     │   │        └── 認証NG → /admin/login へリダイレクト
//     │   └── NO ↓
//     │
//     └── その他 → そのまま通過
```

## 3. public-site/ — 公開サイト（SSG）

一般ユーザー（お客様）に見せる公開ページです。  
SSG（Static Site Generation）で事前にHTMLを生成し、Cloudflare Pages でホスティングします。

```
public-site/
├── 📂 app/
│   ├── layout.tsx               ← 公開サイト共通レイアウト
│   ├── page.tsx                 ← トップページ
│   ├── globals.css              ← グローバルスタイル
│   └── 📂 ssg-shell/           ← SSG シェル（静的HTML生成用）
│       ├── page.tsx             ←   静的エントリーポイント
│       └── client-home.tsx      ←   クライアント側でデータ取得
│
├── 📂 components/               ← 公開サイト用コンポーネント
│   ├── product-card-simple.tsx  ←   商品カード（簡易版）
│   ├── product-detail-modal.tsx ←   商品詳細モーダル
│   ├── product-masonry.tsx      ←   Masonry レイアウト
│   ├── profile-card.tsx         ←   プロフィールカード
│   ├── profile-header.tsx       ←   プロフィールヘッダー
│   ├── recipe-display.tsx       ←   レシピ表示
│   ├── public-nav.tsx           ←   公開サイトナビゲーション
│   └── 📂 ui/                  ←   共通UI部品
│
├── 📂 lib/
│   ├── api-client.ts            ← 公開 API クライアント
│   ├── image-url.ts             ← 画像 URL ヘルパー
│   ├── image-usecases.ts        ← 画像ユースケース
│   └── utils.ts                 ← ユーティリティ
│
├── 📄 package.json
└── 📄 next.config.mjs
```

```
💡 SSG シェルパターンとは？
──────────────────────────
このプロジェクトでは「SSG シェル + CSR」パターンを使っています。

1. ビルド時: 空のHTMLシェル（外枠）だけを静的生成（SSG）
2. ブラウザ: JavaScript が動いた後、Worker API からデータを取得（CSR）
3. 画面表示: 取得したデータで動的にコンテンツを表示

メリット: Cloudflare Pages の無料枠でもデプロイできる
デメリット: 初回表示時にローディング状態が見える
```

## 4. public-worker/ — 公開 API（Hono + CF Workers）

メインの API サーバーです。Hono フレームワークを使い、Cloudflare Workers 上で動作します。

```
public-worker/
├── 📂 src/
│   ├── index.ts                 ← Worker エントリーポイント（Hono アプリ定義）
│   ├── supabase.ts              ← Supabase クライアント生成
│   ├── 📂 routes/
│   │   └── 📂 public/           ← 公開 API ルート群
│   │       ├── index.ts         ←   ルート登録
│   │       ├── products.ts      ←   /api/products
│   │       ├── collections.ts   ←   /api/collections
│   │       ├── recipes.ts       ←   /api/recipes
│   │       ├── profile.ts       ←   /api/profile
│   │       ├── tags.ts          ←   /api/tags
│   │       ├── tag-groups.ts    ←   /api/tag-groups
│   │       └── site-settings.ts ←   /api/site-settings
│   ├── 📂 services/
│   │   └── 📂 public/           ← ビジネスロジック層
│   └── 📂 middleware/
│       └── cors.ts              ← CORS ミドルウェア
│
├── 📄 package.json              ← 依存関係
├── 📄 wrangler.toml             ← Cloudflare Workers 設定
└── 📄 tsconfig.json             ← TypeScript 設定
```

### wrangler.toml の設定例

```toml
name = "public-worker"
main = "src/index.ts"
compatibility_flags = ["nodejs_compat"]

# R2 バケットのバインディング
[[r2_buckets]]
binding = "IMAGES_BUCKET"
bucket_name = "images"

# 環境変数（Secrets で管理するものは含まない）
[vars]
IMAGES_DOMAIN = "https://images.shirasame.com"
```

```
💡 Workers のルート構成パターン
─────────────────────────────
public-worker では「ルート層」と「サービス層」を分離しています。

routes/  → HTTP リクエストの受付（パラメータ解析・バリデーション・レスポンス整形）
services/→ 実際のビジネスロジック（DB 操作・データ加工）

この分離により:
- ルート層はHTTPに関する処理に集中
- サービス層は純粋なビジネスロジックに集中
- テストがしやすくなる
```

### admin-api-proxy の役割

```
問題:
  管理画面 (admin.shirasame.com) と API (public-worker) が
  別のドメインにある → ブラウザの Same-Origin Policy で
  Cookie が送信されない！

解決策:
  admin.shirasame.com/api/* を同一ドメインの Worker で受け取り、
  Cookie を中継して public-worker に転送する

     ブラウザ                  admin-api-proxy           public-worker
  (admin.shirasame.com)   (admin.shirasame.com/api/*)   (Workers)
         │                        │                        │
         │── /api/products ──────>│                        │
         │   (Cookie 付き)        │── 転送（認証情報含む）──>│
         │                        │<── レスポンス ──────────│
         │<── レスポンス ─────────│                        │
```

## 6. shared/ — 共有コード

複数のアプリ（admin-site, public-site, public-worker）で共通して使うコードを配置します。

```
shared/
├── 📂 lib/
│   └── image-usecases.ts        ← 画像 URL 生成の共有ロジック
│                                   - r2KeyToPublicUrl(): R2キー → 公開URL
│                                   - buildResizedUrl(): リサイズ URL 生成
│                                   - getImageSrcSet(): srcset 生成
│
├── 📂 normalize/
│   └── normalizePin.ts          ← レシピピンデータの正規化
│                                   （旧/新スキーマの互換変換）
│
└── 📂 types/
    ├── index.ts                 ← 主要な型定義
    │                               Product, Collection, Recipe,
    │                               User, Tag, ProductImage など
    ├── api.ts                   ← API レスポンスの型
    │                               ApiResponse<T>, PaginatedResponse<T>
    └── recipe.ts                ← レシピ専用の型
                                    PublicRecipeImage, PublicRecipePin
```

```
💡 なぜ shared/ を作るのか？
──────────────────────────
同じ「商品」の型定義を admin-site と public-worker の両方で
別々に書くと、一方だけ変更してもう一方を忘れるというバグが起きます。

shared/ に共通の型やユーティリティを置くことで:
  ✅ 型定義が一箇所にまとまる（Single Source of Truth）
  ✅ 変更が全アプリに反映される
  ✅ コードの重複が減る
```

### 共有型定義の例

```typescript
// shared/types/index.ts
export interface Product {
  id: string;
  userId: string;
  title: string;
  slug: string;
  price: number;
  published: boolean;
  images: ProductImage[];
  tags: string[];
  shortDescription?: string;
  body?: string;
  affiliateLinks?: AffiliateLink[];
}

export interface ProductImage {
  id: string;
  url: string;
  basePath?: string;
  width?: number;
  height?: number;
  role?: 'thumbnail' | 'main' | 'secondary' | 'attachment';
}

// shared/types/api.ts
export interface ApiResponse<T> {
  data: T;
  error?: string;
}

export interface PaginatedResponse<T> {
  data: T[];
  meta: {
    total: number;
    limit: number;
    offset: number;
  };
}
```

```
└── 2026-01-02-add-is-immutable-and-product-type.sql ← 商品タイプ追加
```

```
💡 マイグレーションとは？
──────────────────────
マイグレーションとは、データベースのテーブル構造（スキーマ）を
変更するための手順書です。SQL ファイルとして管理します。

ファイル名規則: YYYY-MM-DD-説明.sql
  例: 2025-12-24-add-affiliate-links-to-products.sql

このプロジェクトでは Supabase のダッシュボードから
SQL エディタで手動実行する方式を採用しています。
```

### マイグレーション SQL の例

```sql
-- migrations/2025-12-24-add-affiliate-links-to-products.sql

-- アフィリエイトリンクテーブルの作成
CREATE TABLE IF NOT EXISTS affiliate_links (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  product_id UUID NOT NULL REFERENCES products(id) ON DELETE CASCADE,
  url TEXT NOT NULL,
  label TEXT,
  provider TEXT,           -- 'amazon', 'rakuten' など
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()
);

-- インデックス
CREATE INDEX idx_affiliate_links_product_id ON affiliate_links(product_id);
```
